# Pre-beta audit WB Profit (v0.7.7.34)

*Сгенерировано 2026-05-28 фоновым агентом — детальный обход dashboard.html, api/*, login.html, index.html.*

---

## КРИТИЧНОЕ (нужно починить до раздачи промокодов)

1. **Возможна подделка тарифа из браузера.** `dashboard.html:2768` — фронт делает `sb.from('profiles').update({plan:'free', plan_expires_at:null})` напрямую. Если RLS на `profiles` разрешает юзеру UPDATE своих строк по всем полям, он через DevTools может присвоить себе `plan:'business', plan_expires_at:'2099-01-01'` навсегда. **Нужно проверить миграции и закрыть.**

2. **Онбординг новичка ломается молча.** `dashboard.html:3087-3098` (`onbFinish`) — после ввода токена сохраняет его только в `localStorage`, не создаёт запись в `wb_accounts`. Следующий вызов `/api/snapshots` возвращает 412 `NO_WB_ACCOUNT`, фронт код 412 не обрабатывает. Юзер получает белый экран после онбординга. Фикс — вызвать создание `wb_accounts` перед редиректом на Главную.

3. **Обход промокод-гейта.** `login.html:204-227` — `signUp` создаёт аккаунт до проверки промокода. Через DevTools юзер может вызвать `sb.auth.signUp(...)` напрямую, минуя форму с промокодом. Также если `apply_beta_code` упадёт после `signUp`, аккаунт создан, но без статуса — `console.warn` молча проглатывает ошибку. Фикс — атомарный server-side endpoint `validate + create + apply`.

4. **Утечка данных между юзерами на одном компьютере.** `dashboard.html:3256-3260` — `logout()` очищает только `wb_token`. Остаются `wb_costs`, `wb_seller_name`, `wb_personal_expenses`, `calc_presets`, `wb_cache_processed` и ещё ~13 ключей. Второй селлер на том же ПК увидит данные первого. Фикс — `localStorage.clear()` или явный список.

5. **PostgREST-инъекция в поиске.** `api/calcs.js:54` — параметр `q` подставляется в `.or(...)`-фильтр без эскейпинга. Юзер может прислать `q=x),name.eq.target` и вытащить чужие данные или сломать запрос. Аналогично в `api/wb-tariffs.js:97` (но этот публичный, риск ниже).

6. **«Только один логгер ошибок».** В `v0.7.7.33` заявлено «глобальный лог JS-ошибок», но `installFrontendErrorLogger` есть только в `dashboard.html`. На `login.html` и `index.html` его нет — ошибки на самом важном экране (регистрация/вход) **не логируются**.

7. **Лендинг привирает.** `index.html:851` — «**230+ селлеров уже считают реальную прибыль»**. Реальность — проект в инвайт-бете v0.7.7.x. Если первый юзер заметит — мгновенная потеря доверия.

8. **«AI советник» не существует.** `index.html:934, 977, 1050` и `dashboard.html:2128` — продаётся «AI советник» с примером совета. В коде **нет ни одного вызова LLM API**. Юзер за 990 ₽/мес получит карту-калькулятор без AI. Юридически — недостоверная реклама.

9. **CSP блокирует Supabase на лендинге.** `vercel.json:41` — `script-src 'self' 'unsafe-inline'` без `cdn.jsdelivr.net`. Supabase SDK на лендинге не загружается. Кнопка «Мой дашборд» для залогиненного юзера на проде **не работает**.

10. **API audit-log не пишет.** `api/profile.js:117,247` — `audit(...)` вызывается с позиционными аргументами, а функция в `lib/audit-log.js:44` ждёт объект. Добавление и удаление WB-кабинетов **молча не пишутся в журнал**.

11. **Service_role используется как anon.** `api/profile.js:347-348` — комментарий «используем service_key как auth», это **service_role** (суперюзер БД). `signInWithPassword` работает «случайно». Любой баг в обработке `body.password` может стать дырой. Фикс — создать отдельный anon-client с `SUPABASE_PUBLISHABLE_KEY`.

## ВАЖНОЕ (надо успеть на первой неделе беты)

12. **Тупик «Оплатить» в Trial/Expired.** `dashboard.html:1960-1975, 3647` — баннеры зовут оплатить, в Настройках единственная реакция — `alert('Платежи через ЮKassa подключим в ближайшем релизе')`. Юзер с истёкшим trial — в тупике. Фикс — переписать в «Бесплатно весь период беты».

13. **XSS в админке.** `dashboard.html:3462-3470` — `innerHTML` с прямой интерполяцией `u.email, p.code` без `escapeHtml`. Low severity, видит только админ — но стоит закрыть.

14. **Себестоимости мигрируют между WB-кабинетами.** `dashboard.html:4360-4438` — `wb_costs` в localStorage хранится под одним ключом. При переключении кабинетов через `switchWbAccount` себестоимости из кабинета №1 попадают в кабинет №2. Фикс — ключ должен включать `wb_account_id`.

15. **Версия захардкожена.** `api/config.js:133` отдаёт `v0.7.7.19`, реальная — `v0.7.7.34`. Мониторинг показывает старое.

16. **CHANGELOG отстал.** В файле последняя запись `v0.7.7.33`, в коде уже `v0.7.7.34`. Юзер увидит несоответствие версии в сайдбаре и в публичном CHANGELOG.

17. **Service_role как fallback на ANON.** `api/plans.js:6-8`, `api/wb-tariffs.js:18` — если `ANON_KEY` отсутствует, используется `SERVICE_ROLE_KEY`. На публичном endpoint. Сейчас ключ клиенту не возвращается, но дисциплина потеряна — должно быть «500 если ANON_KEY нет».

18. **FAQ обещает 12 месяцев данных, реальность другая.** `index.html:189` — «Данные за последние 12 месяцев». На free/expired = 7 дней, на Start = 30. **Не соответствует тарифам.**

19. **FAQ обещает шифрование, которого нет.** `index.html:156` — «Данные хранятся в зашифрованном виде». В коде нет шифрования (WB-токен в `wb_accounts.wb_token` — plaintext). **False claim в публичной FAQ.**

20. **«Обновите страницу» после промокода.** `dashboard.html:3431` — `applyPromo()` показывает «Обновите страницу». UX-провал, должен быть auto-reload.

21. **Badge «Free» в системе без Free.** `dashboard.html:3194, 3208, 3469` — UI всё ещё знает про «Free», хотя в v0.7.7.22 тариф удалён. Юзер с истёкшим тарифом увидит «Free» вместо «Истёк».

23. `ooovsktrans@mail.ru` в 9 местах — вынести в одну константу для будущей замены.

24. Дублированные meta-теги Cache-Control в `dashboard.html:12-21`.

25. Мёртвая функция `saveTokenManual` (`dashboard.html:3731-3751`) — кнопки нет, поле legacy.

26. 37 `console.log/warn/error` в проде — без Sentry засоряет devtools.

27. `console.warn('apply_beta_code error')` в `login.html:222` — молча проглатывает ошибку, юзер не узнает.

28. `document.getElementById('nav-promo')` в трёх местах — элемента нет, активная подсветка мертва.

29. `alert('Аккаунт удалён. До свидания.')` — холодный системный алёрт на финальном экране удаления. Лучше тост.

30. `confirm('Выйти из аккаунта?')` — старомодно. Можно тостом с «Отменить».

31. BETA-полоска обрезается на экранах <340px (есть `text-overflow:ellipsis`).

32. Demo-режим использует `'demo-jwt-placeholder'` — если случайно вызовется backend, 401 будет молчаливый.

33. FBW vs FBO в селекторе фильтра — два разных термина для одного, юзер может запутаться.

34. `alert('Спасибо за фидбек!')` — fallback на alert, если `showToast` не определён.

35. Подсказка про личные расходы говорит «хранится только в браузере» — для юзера с двумя браузерами это «вводи дважды», стоит упомянуть.

36. `calcProfit()` на лендинге считает «прибыль = 69% от выплаты» хардкодом. Для селлера с ДРР 30% — расходится с реальностью.

## Что работает хорошо (важно сохранить)

- `lib/plan-check.js` — централизованная проверка тарифа с кешем. Грамотно.
- `lib/wb-account.js` — `resolveWbAccountId` проверяет принадлежность через `eq('user_id', userId)`. Защита от подделки.
- `api/promo.js` — `userId` берётся ТОЛЬКО из JWT, не из body.
- `vercel.json` — CSP, HSTS, X-Frame-Options, Permissions-Policy на месте.
- `installFrontendErrorLogger` защищён флагом `__errLoggerInstalled` от двойной установки.
- `loadAdminBetaViews` (`dashboard.html:3506-3528`) — универсальный рендер для 4 view, один хелпер `renderAdminTable`. Чисто.
- Удаление аккаунта — двойная защита (пароль + слово DELETE), audit-log, атомарная RPC `delete_user_account`.
- `api/wb.js:159` — WB-токен в БД не хранится, для rate-limit используется sha256-хеш. Правильно.
